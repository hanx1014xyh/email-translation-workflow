# Email Translation Workflow

一个基于 n8n 的邮箱监听与自动翻译工作流，支持英文邮件自动翻译为中文并推送通知，同时支持用中文回复并以英文发出，实现跨语言邮件通信。

## 项目架构

本项目包含两条核心链路，共同实现"接收英文邮件 → 中文通知 → 中文回复 → 英文发送"的完整跨语言通信流程。

### 链路一：英文邮件接收 → 中文通知推送

Email Trigger (IMAP) → 英文转中文 (DeepSeek) → Switch路由 → 企微/飞书/钉钉机器人通知


- **邮箱监听**：通过 IMAP 协议实时监听 Gmail 收件箱，自动获取新邮件
- **智能翻译**：调用 DeepSeek 大语言模型，将英文邮件的主题和正文翻译为中文
- **通知内容**：推送内容包含发件人（姓名+邮箱）、邮件主题、翻译后的正文，用户一眼即可了解邮件全貌
- **多渠道路由**：通过 Switch 节点实现渠道路由，支持企业微信、飞书、钉钉；未指定渠道时默认走企业微信（Fallback机制）

### 链路二：中文回复 → 英文邮件发送

Webhook (接收中文回复) → 中文转英文 (DeepSeek) → Send Email (SMTP发送)


- **接收回复**：通过 Webhook 接收用户的中文回复内容及收件人信息
- **智能翻译**：调用 DeepSeek 将中文回复翻译为英文
- **邮件发送**：通过 SMTP 将翻译后的英文内容发送给指定收件人

## 整体流程图

### 链路一：接收与通知

[Gmail收件箱] → [IMAP监听] → [DeepSeek英转中] → [Switch路由] → [企微 / 飞书 / 钉钉 机器人]


### 链路二：回复与发送

[中文回复] → [Webhook] → [DeepSeek中转英] → [SMTP发送] → [原始发件人收到回复]


## 通知消息示例

用户在机器人中收到的通知格式如下：

📩 新邮件通知 发件人：Zhang San zhang@company.com 主题：Meeting Tomorrow 正文：你好，我想与您安排明天下午3点开会，讨论项目进展情况。


## 多渠道扩展性设计

通知推送采用 Switch + HTTP Request 的架构设计，通过统一的消息路由机制实现多渠道扩展：

| 渠道 | Webhook URL格式 | 消息体格式 |
|------|----------------|-----------|
| 企业微信 | `https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key={KEY}` | `{"msgtype":"text","text":{"content":"..."}}` |
| 飞书 | `https://open.feishu.cn/open-apis/bot/v2/hook/{TOKEN}` | `{"msg_type":"text","content":{"text":"..."}}` |
| 钉钉 | `https://oapi.dingtalk.com/robot/send?access_token={TOKEN}` | `{"msgtype":"text","text":{"content":"..."}}` |

扩展新渠道只需两步：
1. 在 Switch 节点添加新的路由规则
2. 添加对应的 HTTP Request 节点配置 Webhook URL 和消息格式

## 技术栈

- **工作流引擎**：n8n（可视化自动化平台）
- **邮件协议**：IMAP（收件）/ SMTP（发件）
- **翻译模型**：DeepSeek Chat API（兼容 OpenAI 格式）
- **通知渠道**：企业微信 / 飞书 / 钉钉 Webhook 机器人

## 使用方法

### 前置条件

- n8n 实例（Cloud 或 Self-hosted）
- Gmail 账号（需开启 IMAP 和应用专用密码）
- DeepSeek API Key
- 企微/飞书/钉钉机器人 Webhook URL（按需配置）

### 部署步骤

1. 在 n8n 中导入 `Email Translation Workflow.json`
2. 配置 IMAP credential（Gmail 邮箱 + 应用专用密码）
3. 配置 SMTP credential（Gmail 邮箱 + 应用专用密码）
4. 配置 DeepSeek API Key（Header Auth: `Authorization: Bearer {YOUR_KEY}`）
5. 将通知节点中的 Webhook URL 替换为真实的机器人地址
6. 将 Send an Email 节点中的 fromEmail 替换为你的 Gmail 地址
7. 激活 workflow

## 当前版本说明与设计思考

### 已实现

- 英文邮件自动监听与中文翻译（含主题和正文）
- 翻译结果通过机器人推送通知，包含发件人、主题、正文完整信息
- 多渠道路由架构（Switch + Fallback 默认走企微）
- 中文内容翻译为英文并通过邮件发送

### 当前限制与原因

由于没有真实的企业微信/飞书/钉钉机器人环境，以下功能以 Webhook 模拟实现，但产品方案已设计完整：

**1. 回复路径的闭环**

当前用户需要通过 Webhook 接口手动提交回复内容和收件人地址。在接入真实机器人后，完整的体验应该是：用户在机器人对话框中直接用中文回复，机器人平台通过回调接口自动将回复内容 POST 到 n8n 的 Webhook，后续翻译和发送流程自动完成，用户无需任何额外操作。

**2. 上下文记忆与自动匹配收件人**

当前两条链路之间没有共享的"记忆"。理想方案是：链路一收到邮件时，将发件人地址、邮件主题、邮件ID等信息存入数据库（如 MySQL 或 Airtable）；链路二收到回复时，根据邮件编号从数据库中查出原始发件人地址，自动填入收件人字段。这样用户回复时完全不需要手动输入对方邮箱。

**3. 批量邮件的区分与指定回复**

当同时收到多封邮件时，每封通知应携带唯一编号（如 #001、#002）。用户回复时带上编号（如"#001 好的，明天见"），系统根据编号匹配对应邮件的发件人，实现精准回复。

**4. 基于自然语言的智能回复匹配（进阶方案）**

编号回复虽然可靠，但仍然要求用户记忆和输入编号，体验不够自然。更理想的方案是让 LLM 承担意图识别的角色，用户可以用完全自然的语言描述回复对象，系统自动理解并匹配。例如：

- 按姓名回复："回复张三那封邮件：好的，我准时参加。"
- 按公司回复："回复 Google 发来的邮件：我们同意这个方案。"
- 按时间回复："回复今天上午那封邮件：收到，我会尽快处理。"
- 批量回复："把今天收到的三封邮件统一回复：已收到，稍后逐一回复。"

实现方式：用户的回复内容先经过一个 LLM 意图解析节点，提取出回复对象（姓名/公司/时间等维度）、回复范围（单封/批量）和回复正文，再根据提取结果去数据库中匹配对应邮件，最后完成翻译和发送。这样 LLM 在整个产品中不仅承担翻译功能，还承担了意图识别和信息提取的角色，真正实现"让机器理解人"而非"让人适应机器"。

## 后续优化方向

- **多轮对话**：基于邮件 Message-ID 和 In-Reply-To 头部实现邮件线程关联，支持多轮往返通信
- **HTML邮件支持**：当前处理纯文本邮件，后续可扩展支持 HTML 格式邮件的解析与翻译
- **错误处理**：添加翻译失败重试、邮件发送失败告警、API 调用超时等异常处理机制
- **语言自动检测**：自动识别邮件语言，非英文邮件跳过翻译直接通知，避免无效翻译
- **翻译质量优化**：针对专业领域（如法律、技术）定制翻译 prompt，提升翻译准确度
